Java基础

ClassLoader（类加载机制）

Java类都需要经过JVM加载才能运行，ClassLoader的主要作用就是Java类文件的加载

ClassLoader类有如下核心方法：

  1. loadClass（加载指定的Java类）
  2. findClass（查找指定的Java类）
  3. findLoadedClass（查找JVM已经加载过的类）
  4. defineClass（定义一个Java类）
  5. resolveClass（链接指定的Java类）

URLClassLoader

URLClassLoader继承了ClassLoader，且提供了加载远程资源的能力，在写漏洞利用的payload或者webshell的时候可以加载远程的jar来实现远程的类方法调用

BCEL ClassLoader

BCEL是一个用于分析、创建和操纵Java类文件的工具库，其类加载器在解析类名时会对ClassName中有$$$BCEL$$$标识的类做特殊处理，这一特性可以被用于Payload编写。影响于BCEL<6.0

Java反射

反射机制是Java非常重要的动态特性，可以获取类的成员方法(Methods)、成员变量(Fields)、构造方法(Constructors)等信息，还可以动态创建Java类实例、调用类方法、修改类成员变量值等。

获取Class对象

没有Class对象Java就没有地方操作反射，所以需要先获取Class对象

  1. className.class
  2. Class.forName(className)。
  3. classLoader.loadClass(className);

数组类型和Runtime类稍有不同

反射Runtime

java.lang.Runtime中有个可以执行本地命令的exec方法，在payload中可以用于执行本地系统命令，而Runtime的构造方法往往是私有的，无法直接Runtime rt = new Runtime();,但反射可以做到在此情况下调用构造方法，即通过反射去修改方法的访问权限(constructor.setAccessible(true))。

此外，关于构造方法的获得，runtimeClass1.getDeclaredConstructor和runtimeClass1.getConstructor都可以获取到构造方法，区别在于后者无法获取到私有方法，所以前者使用居多。

获得对象后，可以进一步调用其中的方法和成员，和构造方法类似，带Declared可以拿到更多，不带的拿到更少。

方法调用使用method.invoke

没有修改成员变量的权限时可以使用: field.setAccessible(true);而final修饰的成员变量的修改需要先修改方法（感觉这里怪怪的，大致意思是要先把final去掉？）

sun.misc.Unsafe

sun.misc.Unsafe是Java底层API(仅限Java内部使用反射可调用)提供的Java类，提供了底层的内存、CAS、线程调度、类、对象等操作、提供的几乎所有的方法都是不安全的

allocateInstance

大概就是无法直接通过反射区创建某个类的实例，那么可以考虑用这个Unsafe下的方法去绕过限制

defineClass

ClassLoader可以在JVM中注册一个类,Unsafe的defineClass方法也有同样的功能

Java文件系统安全

Java SE中内置了两类文件系统：java.io（基于阻塞模式的IO的文件系统）和java.nio（JDK7+基于NIO.2的文件系统。）

Java IO 文件系统

Java有个抽象类java.io.FileSystem，抽象了对文件的操作，不同的操作系统有不一样FileSystem,例如 java.io.UnixFileSystem、java.io.WinNTFileSystem。

其底层会调用动态链接库中c实现的java Native方法

不是所有的文件操作都在java.io.FileSystem中定义

Java NIO.2 文件系统

和Java IO是完全独立的，关于文件封装等倒是和IO那边一样，比较显而易见的是有些WAF/RASP只防御了IO，那显然可以通过NIO.2搞点事情

文件读写

  • FileInputStream（读文件）
  • FileOutputStream（写文件）
  • RandomAccessFile（可以读写文件，类中readxxx方法均在最终调用read0或readBytes方法）
  • FileSystemProvider（nio文件系统下的，可以读写）
  • 其余大部分api都是源自上面几个的封装
