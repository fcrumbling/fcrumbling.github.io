小技巧

实体快速创建

关于实体的创建，因为我拿navicat建了表，所以找了找可以用数据库的表生成实体的方法。

大致步骤：选中表格-Tools-Scripted Extensions-Scripted Extensions-选中目标文件夹

需要自己写注解和改package，无需写set/put

注解参照之前写博客后端时用的，lombok+jpa的注解：

@Entity 是一个 JPA 注解，用于将一个普通的 Java 类标识为 JPA 实体类，表示这个类在数据库中对应一张表。 @Entity 注解使 Spring Data JPA 就能够根据实体类的定义自动生成对应的数据表，并进行数据库操作。

@Entity 注解通常与 @Table 注解一起使用，@Table 注解用于指定实体类对应的数据库表的名称，如果省略 @Table 注解，则默认使用与类名相同的表名。

需要通过各种注解来描述实体类与数据库表之间的映射关系，比如 @Id 表示该属性是主键，@GeneratedValue 表示该属性的值是自动生成的，@Column 表示该属性与数据库表的列之间的映射关系等等。

Mapper/Repository，与Service,ServiceImpl的关系

Mapper/Repository是用于定义数据持久化操作的接口，定义了对数据库的增删改查等操作，在 Spring Data JPA中叫Repository，可以通过继承 JpaRepository 或 CrudRepository 来实现对数据库的操作

Service 是业务逻辑层，它包含了应用程序的业务逻辑。Service 组件负责处理业务逻辑，通常会调用一个或多个 Mapper（Repository）来完成数据持久化操作【这个有点没看出来】。Service 组件的作用是将 Controller 层接收到的请求转化为对应的业务逻辑，并调用相关的 Mapper 完成数据的持久化操作。

ServiceImpl 是 Service 接口的具体实现类。它实现了 Service 接口定义的业务逻辑，并调用相关的 Mapper（Repository）来完成数据的持久化操作。

Service则是写业务逻辑（其实就是定义函数）

ServiceImpl是函数实现

让我这个野路子比较费解的其实是Mapper/Repository，直观来说Mapper/Repository接口只需要继承相关依赖里的数据库操作，或者进行自定义数据库操作

orElse(null)

来自JpaRepository的findbyid方法会返回一个Optional类

而orElse是一个 Optional 类的方法，它表示如果 Optional 对象中存在值，则返回该值，否则返回指定的默认值,这里是 null,

Optional 则是 Java 8 中引入的一个类，用于表示一个可能为null的值。它的设计目的是为了避免空指针异常

报错解决

测试第一个接口的时候经典的404了

问题解决：Application需要位于controller文件夹同层

紧接着遇到了500

报错是各种列名xx无效

解决方案，数据库伏笔了，应该全体小写，其中实例类中间的大写对应下划线；写注解应该也行吧？

另外要注意的是本来用来做备用字段的other删除了，因为这玩意儿为空会有同样的500错误

解决方案，在idea里拿sql语句去跑了，发现无法识别表名，我的数据库结构是库-schema-table，所以我把注解改成了：

@TableName("fwwb_schema.exhibitor")

也就是加了前缀模式，终于解决了，困扰了一天多，还错怪了jpa，啊哈哈。总而言之，可以步入正轨开发业务了，真不容易。
